# 数識電収 リファクタリング計画

## 目的

数識電収の現在の安全性、公開URL、操作方法を保ったまま、コードを分割し、
LuaLaTeX、HTML、Typst、全文検索、原稿関係図、読書経路などを独立して追加・保守
できる構成にする。

この文書は人間とAIの両方が作業を引き継ぐための基準である。リファクタリングを
始める前と各段階の完了時に読み直すこと。

## 2026年7月時点の状態

- 公開記事は191件。
- `scripts/paper_tool.py` は約320行の互換CLIとなり、HTML生成、公開配置、SHA保護、
  個人情報検査、変更承認、四種の取り込みを専用モジュールへ分離済みである。
- `scripts/migration_ledger.py` は18行の互換窓口となり、約2,000行の台帳本体と
  非公開確認ページを `tools/legacy_migration/` へ隔離済みである。
- 自動テストは52件あり、手元では成功している。
- 191記事のSHA検査、カタログ検査、台帳231件の検査、公開サイト生成、リンク検査は
  成功している。
- GitHub Actionsは単体テスト、SHA、カタログ、台帳、公開サイトの基準スナップ
  ショットを検査する。
- 移行は完了しているため、台帳機能は今後のサイト機能から隔離できる。

## 絶対に守る条件

1. `python3 scripts/paper_tool.py ...` という既存の操作方法を維持する。
2. 明示的な指示なしに191記事のTeX、PDF、BibTeX、BST、図版を変更しない。
3. 保護対象ファイルのSHA-256を意図せず変更しない。
4. `paper.json` を原稿情報の正本として維持する。
5. 公開URLと旧slugの互換URLを変更しない。
6. リファクタリングだけの段階では、公開内容と見た目を原則として変更しない。
7. リファクタリングと新機能追加を同じコミットに混ぜない。
8. 各コミットを単独でテスト・デプロイ可能にする。
9. 生成失敗時に未検査ファイルを公開しない。
10. 外部変換で生成したHTMLやTypstは派生物として扱い、原稿を上書きしない。

- [x] ブログ本文のみ、PDFあり、TeXのみ、ビルド対象のファイル条件を検査する。
- [x] 承認履歴の対象ファイルと個人情報検査記録の現在SHAを検査する。

`StageFeature` として登録し、サイト本体とは別の一時領域へ出力する。既存ファイルとの
衝突を拒否し、成功時だけ本体へ統合するため、失敗した変換の途中ファイルは公開されない。
`FeatureResult` は機能名、必須・任意、対象原稿slug、成否、失敗理由を保持する。

サイト本体も公開先と同じ親ディレクトリの一時領域へ作る。SHA、生成、配置、追加機能、
リンク検査がすべて成功してから完成先へ置き換える。途中で失敗した場合は一時領域だけを
削除し、すでに存在する完成先は変更しない。従来の
`python3 scripts/paper_tool.py stage _site` という操作方法と公開物は変わらない。

完了条件：各段階を単独テストでき、失敗時に途中の公開物を採用しない。

## 第6段階：取り込み、個人情報検査、SHA保護の分離

TeXのみ、PDFのみ、通常、ブログ本文のみの取り込みに共通する処理をまとめる。

進捗：

- [x] 保護対象ファイルのSHA検査を `protection/hashes.py` へ分ける。
- [x] TeX・PDFの検査、検査票、手動確認、強制理由を `protection/privacy.py` へ分ける。
- [x] 明示的変更のSHA・個人情報記録更新を `protection/approval.py` へ分ける。
- [x] 保存先決定、バイト単位コピー、コピーSHA確認、manifest完成を
  `importing/common.py` へまとめる。
- [x] TeXのみ、PDFのみ、通常・ブログ本文のみの取り込みを個別モジュールへ分ける。
- [x] 取り込み失敗時に作成途中の原稿フォルダを削除する処理を共通化する。
- [x] 検査未実施、検査SHA不一致、強制理由なし、確認方法混用を単体テストする。
- [x] PDF検査画像欠落、失敗検査の強制記録、コピー失敗時の後始末を単体テストする。
- [x] 複数公開TeX・PDFの検査漏れと既存四種取り込みを統合テストで確認する。
- [x] 既存CLI、全50テスト、191記事、台帳、リンク、公開物の一致を確認する。

```text
入力検査
  -> 個人情報検査記録の確認
  -> 保存先決定
  -> バイト単位コピー
  -> コピー前後SHA確認
  -> paper.json作成
  -> カタログ再生成
```

安全性が最重要なので、正常系だけでなく、検査未実施、SHA不一致、強制理由なし、
複数ファイルの検査漏れなどの失敗系を先にテストする。

CLIは従来どおり `inspect-file`、`import-tex`、`import-pdf`、`import-paper`、
`approve-change` を提供する。処理本体はリポジトリパスと個人情報検査票の保存先を
引数として受け取るため、テスト用環境でも実原稿へ触れず直接検査できる。

すべての取り込みは、入力と個人情報検査票を確認してから新しい保存先を作り、原稿を
バイト単位でコピーする。コピー元とコピー先のSHA-256が一致した後だけ `paper.json` と
`keywords.txt` を作り、manifest検査と保護ファイル検査を通す。途中で一つでも失敗した
場合は保存先全体を削除する。すでに存在する原稿フォルダは引き続き上書きしない。

通常確認と強制確認は同時指定できず、強制確認には空でない理由が必要である。PDFの
通常確認には全ページ画像が必要で、検査票のSHAが現在ファイルと違う場合は再検査を
要求する。変更承認では、指定外の保護ファイルに未承認変更があれば停止する。

完了条件：既存の安全規則が弱まらず、重複処理が共通化される。

実装後も、互換CLIとCIの台帳検査は維持する。約430行の確認ページはlegacy専用生成器で
管理し、新機能側から台帳CSV・JSONを直接参照しない。

台帳を全面的に作り直すのではなく、凍結後も必要時に保守できる状態を目指す。

利用者向けコマンドと台帳ファイルの場所は変更しない。既存のREADME、手順書、
GitHub Actionsは引き続き `python3 scripts/migration_ledger.py ...` を使用する。互換窓口が
legacy実装の `main` を呼び出すため、過去の操作例を修正せず利用できる。

`dempa_site/` 以下は `migration_ledger`、`legacy_migration`、`ledger/` を参照しない。
したがって、通常の原稿取り込み、カタログ生成、公開サイト生成、新機能追加は台帳の
CSV・JSON形式や候補照合の内部処理に依存しない。台帳検査だけは移行記録の監査として
CIで継続する。

完了条件：通常のサイト生成が台帳内部実装に依存せず、既存台帳コマンドも動く。

## 第8段階：機能追加用インターフェース

新機能が再びCLIファイルへ集中しないよう、概念上次の共通形式を用意する。

進捗：

- [x] `SiteFeature` の `validate` と `generate` の共通契約を定義する。
- [x] 単純な関数を登録できる `FunctionFeature` を用意する。
- [x] 通常公開の登録先を `features/registry.py` 一か所へまとめる。
- [x] 検査と生成を機能ごとの一時領域で実行し、成功時だけ統合する。
- [x] 必須・任意、有効・無効、原稿slug、失敗段階を結果へ記録する。
- [x] 第5段階の `StageFeature` を互換アダプタとして維持する。
- [x] 無効機能、検査失敗、出力衝突が基本サイトと他機能を止めないことをテストする。
- [x] 新機能の実装・登録・安全規則を `docs/SITE_FEATURES.md` に記録する。
- [x] 既存CLI、全56テスト、191記事、台帳、リンク、公開物の一致を確認する。

```python
class SiteFeature:
    def validate(self, catalog): ...
    def generate(self, catalog, output): ...
```

想定する独立機能：

- 日替わり・ランダム記事
- 読書経路
- 原稿の系譜
- 原稿関係図
- LaTeXML HTML変換
- MathJaxによる数式閲覧
- Pagefind全文検索
- 定理・反例・定義索引
- 訂正・注釈
- Typst変換

単純な機能まで過度に抽象化しない。複数の生成機能が同じ手順を必要とした時点で共通化する。

実装は `dempa_site/features/base.py` に契約、`runner.py` に隔離実行、`registry.py` に
通常公開の登録だけを置く。現在の登録一覧は空なので、この段階だけでは公開物を増減させない。
将来の機能はそれぞれ固有のモジュールで実装し、CLIや公開パイプラインには機能固有処理を
追加せずレジストリへ登録する。

任意機能の事前検査または生成が失敗しても、その途中出力を破棄して次の機能へ進む。
必須機能の失敗時だけ公開全体を中止する。有効・無効は機能単位で設定でき、無効機能は
`validate` も `generate` も呼ばない。実行結果は `generated`、`failed`、`disabled` を持ち、
失敗時は `validation` または `generation` の段階と理由も残す。

完了条件：一つの機能を無効にしても基本サイトと他機能を生成できる。
