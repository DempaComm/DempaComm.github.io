# 数識電収 リファクタリング計画

## 目的

数識電収の現在の安全性、公開URL、操作方法を保ったまま、コードを分割し、
LuaLaTeX、HTML、Typst、全文検索、原稿関係図、読書経路などを独立して追加・保守
できる構成にする。

この文書は人間とAIの両方が作業を引き継ぐための基準である。リファクタリングを
始める前と各段階の完了時に読み直すこと。

## 2026年7月時点の状態

- 公開記事は191件。
- `scripts/paper_tool.py` は約1,300行となり、HTML生成をサイト専用モジュールへ
  分離済みである。原稿検査、SHA保護、個人情報検査、取り込み、公開配置、リンク検査、
  CLIは後続段階でさらに分離する。
- `scripts/migration_ledger.py` は約2,000行となり、433行の確認ページ生成を分離済み
  である。台帳、候補照合、CLIは第7段階で隔離する。
- 自動テストは41件あり、手元では成功している。
- 191記事のSHA検査、カタログ検査、台帳231件の検査、公開サイト生成、リンク検査は
  成功している。
- GitHub Actionsは単体テスト、SHA、カタログ、台帳、公開サイトの基準スナップ
  ショットを検査する。
- 移行は完了しているため、台帳機能は今後のサイト機能から隔離できる。

## 絶対に守る条件

1. `python3 scripts/paper_tool.py ...` という既存の操作方法を維持する。
2. 明示的な指示なしに191記事のTeX、PDF、BibTeX、BST、図版を変更しない。
3. 保護対象ファイルのSHA-256を意図せず変更しない。
4. `paper.json` を原稿情報の正本として維持する。
5. 公開URLと旧slugの互換URLを変更しない。
6. リファクタリングだけの段階では、公開内容と見た目を原則として変更しない。
7. リファクタリングと新機能追加を同じコミットに混ぜない。
8. 各コミットを単独でテスト・デプロイ可能にする。
9. 生成失敗時に未検査ファイルを公開しない。
10. 外部変換で生成したHTMLやTypstは派生物として扱い、原稿を上書きしない。

## 第3段階：paper.jsonの型と検査の分離

辞書を各所で直接参照する方式から、検査済みの型付きデータへ移行する。

進捗：

- [x] `schemas/paper.schema.json` に現行manifestと省略可能な将来項目を定義する。
- [x] 外部依存なしで使用範囲のJSON Schemaを検査する構造検査層を追加する。
- [x] 日時、slug、年、順番、数学分類、Unicode、ビルド条件を意味検査層へ移す。
- [x] ブログ本文のみ、PDFあり、TeXのみ、ビルド対象のファイル条件を検査する。
- [x] 承認履歴の対象ファイルと個人情報検査記録の現在SHAを検査する。
- [x] 関連記事の参照先slugとstatement識別子をコレクション単位で検査する。
- [x] `Paper` と入れ子の型を追加し、三つのCLIを型付きローダーへ接続する。
- [x] 構造違反と代表的な意味違反の単体テストを追加する。
- [x] 既存191manifestを無変更のまま検査し、公開物の完全一致を確認する。

実装は `dempa_site/manifests/` の `loader.py`、`model.py`、`schema.py`、
`validation.py` に分ける。読み込みは必ず「JSON読込 → Schema構造検査 → 意味検査 →
不変の型付きモデル生成」の順で行う。`Paper` は後続のHTML分離まで既存生成コードを
保てるよう、値の複製を返す読み取り専用の辞書互換口を持つ。新しい処理は
`paper.files`、`paper.build` などの型付き属性を使う。

`history`、`corrections`、`statements`、`relations`、`license` は省略可能であり、
既存191個の `paper.json` には追加も再整形も行っていない。これらを実際に表示・生成
する機能は各機能の実装段階で追加する。

候補となる型：

- `Paper`
- `PaperFile`
- `BuildSettings`
- `PrivacyReview`
- `ApprovedChange`
- `HistoryEvent`
- `Correction`
- `Statement`
- `PaperRelation`

JSON Schemaは配列、文字列、必須項目などの構造検査に使う。次のような意味上の条件は
Pythonで検査する。

- 公開日時、同日内番号、slugの一致
- ビルドルートと公開ファイルの対応
- ブログ本文のみ、PDFのみ、TeXのみの条件
- 個人情報検査記録と現在SHAの一致
- 関連記事や読書経路の記事IDが実在すること

既存191個の `paper.json` を一括整形・一括書換えしない。新しい項目は原則として
省略可能にし、必要な場合だけschema versionの更新手順を別途設ける。

完了条件：現在の全manifestが通り、不正な代表例を個別テストできる。

別コミットにする。

完了条件：生成HTMLが同一または承認済みの差だけになり、表示確認も通る。

## 第5段階：公開処理のパイプライン化

現在の `command_stage` を次の独立した段階へ分ける。

```text
原稿とmanifestの検査
  -> メタデータ集計
  -> 静的ページ生成
  -> 公開ファイル配置
  -> RSS・サイトマップ生成
  -> 追加機能生成
  -> リンク検査
  -> 公開物完成
```

追加機能について「失敗したら公開停止する必須機能」と「PDF・TeX表示へ戻せる任意機能」
を区別する。LaTeXML、Typst、Pagefindなどは原稿ごとに成否を記録し、失敗した原稿の
基本ページを失わないようにする。

完了条件：各段階を単独テストでき、失敗時に途中の公開物を採用しない。

## 第6段階：取り込み、個人情報検査、SHA保護の分離

TeXのみ、PDFのみ、通常、ブログ本文のみの取り込みに共通する処理をまとめる。

```text
入力検査
  -> 個人情報検査記録の確認
  -> 保存先決定
  -> バイト単位コピー
  -> コピー前後SHA確認
  -> paper.json作成
  -> カタログ再生成
```

安全性が最重要なので、正常系だけでなく、検査未実施、SHA不一致、強制理由なし、
複数ファイルの検査漏れなどの失敗系を先にテストする。

完了条件：既存の安全規則が弱まらず、重複処理が共通化される。

単純な機能まで過度に抽象化しない。複数の生成機能が同じ手順を必要とした時点で共通化する。

完了条件：一つの機能を無効にしても基本サイトと他機能を生成できる。

## AIへ作業を依頼するときの指示

AIは次を守ること。

1. 作業開始時にこの文書と `docs/MIGRATION.md` を読む。
2. 作業対象外の既存変更を消さない。
3. 保護対象原稿を編集しない。
4. 一度に一段階だけ実施する。
5. 変更前に現在のテストと検査を実行する。
6. ファイル移動では内容変更を混ぜない。
7. 変更後に共通完了条件を検査する。
8. コミット・pushは明示的に指示された場合だけ行う。
9. 出力差が生じた場合は、承認を得ずに正当化して進めない。
10. 不明な点は安全側へ倒し、PDF・TeXの現行公開を維持する。
